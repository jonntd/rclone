# 🔧 115网盘统一QPS管理实现报告

## 🎯 问题背景

### **115网盘的特殊性**
- **全局账户级别限制**：770004错误影响整个账户的所有API调用
- **共享QPS池**：所有API端点共享同一个QPS配额
- **严格反爬机制**：频率过高会触发账户级别的限制

### **之前的问题**
- 多个调速器各自独立，可能导致总QPS超限
- 不同API使用不同频率，难以统一管理
- 缺乏全局QPS协调机制

## 🔧 统一QPS管理方案

### **1. 统一QPS配置**
```go
// 115网盘统一QPS配置 - 全局账户级别限制
unifiedMinSleep      = 250ms  // ~4 QPS - 统一全局限制
conservativeMinSleep = 500ms  // ~2 QPS - 保守模式（遇到限制时）
aggressiveMinSleep   = 150ms  // ~6 QPS - 激进模式（网络良好时）
```

### **2. 简化调速器架构**
```go
type Fs struct {
    globalPacer   *fs.Pacer // 主要调速器 - 4 QPS
    tradPacer     *fs.Pacer // 传统API - 2 QPS (保守)
    downloadPacer *fs.Pacer // 下载API - 4 QPS (统一)
    uploadPacer   *fs.Pacer // 上传API - 2 QPS (保守)
}
```

### **3. 智能调速器选择**
```go
func (f *Fs) getPacerForEndpoint(endpoint string) *fs.Pacer {
    switch {
    case strings.Contains(endpoint, "/open/ufile/files"):
        return f.globalPacer    // 文件列表 - 统一QPS
    case strings.Contains(endpoint, "/open/ufile/upload"):
        return f.uploadPacer    // 上传操作 - 保守QPS
    case strings.Contains(endpoint, "/open/ufile/download"):
        return f.downloadPacer  // 下载操作 - 统一QPS
    default:
        return f.globalPacer    // 默认 - 统一QPS
    }
}
```
